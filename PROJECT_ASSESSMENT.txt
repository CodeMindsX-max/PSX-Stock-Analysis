Date: 2026-04-18

========================================================================
1. EXECUTIVE SUMMARY
========================================================================

This project is an end-to-end PSX KSE100 analysis system. Collectively, it:

- scrapes the latest PSX KSE100 market snapshot from the PSX website
- validates and stores raw data
- merges live data with historical seed data
- cleans and deduplicates the dataset
- generates ML features
- trains a RandomForest prediction model
- saves the trained model with metadata for API use
- serves a Flask API for health, data, insights, prediction, and file management
- serves a responsive dashboard with charts, auto-refresh, and file previews

In plain words, this is not just a model script. It is a small platform:

data ingestion -> data validation -> feature engineering -> model training
-> model packaging -> API -> dashboard -> artifact management

That is ambitious for a student project, and it shows good ownership and
product thinking.

My rating:

- Educational / portfolio value: 8/10
- Current implementation quality: 6.5/10
- Production readiness today: 4/10

Why this rating:

- The project already has real moving parts and many of them work.
- The code shows effort to add validations, modular scripts, testing, and UI.
- The biggest quality gap is not "can it run?" but "is the data/model flow
  trustworthy and deployable long-term?"

The strongest parts:

- real live scraping
- clear pipeline separation
- basic test coverage
- dashboard and API integration
- artifact management and retention

The biggest risks:

- live history is not truly cumulative across runs
- the historical seed data is outdated relative to live data
- the deployed repo will not contain the data/model files it depends on
- destructive endpoints are publicly callable with no authentication
- the current web request triggers a long-running training pipeline synchronously


========================================================================
2. WHAT THE PROGRAM IS DOING COLLECTIVELY
========================================================================

High-level system flow:

1. scripts/fetch_data.py
   Scrapes live KSE100 values from the PSX website, validates the numbers,
   and saves a timestamped raw CSV plus latest_live_data.csv.

2. scripts/clean_data.py
   Loads the historical seed CSV and the newest live CSV, standardizes the
   schema, converts dates/numbers, removes invalid rows, removes duplicate
   dates, sorts ascending, and writes cleaned outputs.

3. scripts/features.py
   Creates ML features such as Return, MA_7, MA_30, and Volatility, while
   keeping the latest unlabeled row available for API prediction.

4. scripts/train_model.py
   Trains a RandomForestClassifier using the processed feature dataset,
   saves the model, feature columns, latest feature row, metrics, warnings,
   and profile metadata into model.pkl.

5. scripts/run_pipeline.py
   Orchestrates the full workflow from live fetch to model save, archives
   outputs, and prunes old archive files.

6. app.py
   Exposes Flask routes for:
   - /health
   - /data
   - /predict
   - /insights
   - /api/files
   - /api/files/preview
   - /api/files/delete
   - /api/pipeline/run
   and serves the dashboard HTML.

7. static/index.html
   Renders a laptop-first dashboard with charts, status badges, auto-refresh,
   file management, and a pipeline trigger button.

4. FILE-BY-FILE REVIEW
========================================================================

4.1 .gitignore

Assessment:

- Good: keeps local artifacts out of Git.
- Major risk: it ignores data/raw/*, data/processed/*, and *.pkl.
- Result: the repository itself does not track the historical seed data,
  processed datasets, or model files.

Why this matters:

- A fresh deployment from Git will not contain the required seed dataset.
- The app can start, but data/model-dependent features will fail until the
  pipeline is run.
- The pipeline itself currently expects the seed history file to exist.

Observed fact:

- git ls-files confirms that data files and model files are not tracked.

Conclusion:

- This is good for repo cleanliness, but it is a deployment blocker unless
  the project is changed to bootstrap its own data or a tracked seed sample
  is added.


4.2 README.md

Assessment:

- Too minimal for the current complexity of the project.
- Does not explain architecture, setup, running the pipeline, or deployment.

Conclusion:

- Documentation is far below the level the code now deserves.


4.3 manual.txt

Assessment:

- Reads like an earlier learning note rather than current project docs.
- Contains encoding/mojibake issues.
- Does not match the current implementation anymore.

Conclusion:

- Should be replaced by real setup/developer documentation.


4.4 requirements.txt

Good:

- Core runtime packages are present: Flask, pandas, scikit-learn, requests,
  beautifulsoup4.

Weaknesses:

- gunicorn is missing, even though Procfile expects it.

4.5 Procfile

Good:

Weakness:

4.6 app.py

Strengths:

- Clean route layout.
- Good user-facing API surface.
- Health endpoint exists.
- Prediction endpoint uses saved latest_features if available.
- File management routes are practical for a dashboard app.
- Pipeline lock prevents two overlapping runs inside the same process.

Weaknesses:

- Long-running pipeline is executed directly inside a web request.
- Pipeline locking is only process-local, not distributed.
- No authentication or authorization for:
  - /api/pipeline/run
  - /api/files/delete
- Model and data are loaded from disk repeatedly on requests.
- No structured logging; operational issues rely on prints and generic errors.
- Entire architecture depends on local filesystem state.

Professional judgment:

- app.py is good for a local MVP and demo app.
- It is not yet structured like a production Flask service.


4.7 scripts/pipeline_utils.py

Strengths:

- Good separation of reusable filesystem logic.
- Directory constants are centralized.
- Archive path generation is clean.
- Managed file preview and deletion are useful admin features.
- Protected files logic is a strong safety measure.

Weaknesses:

- Storage root is hard-coded to local project directories.
- No environment-based path override.
- Pickle preview loads a pickle directly; pickle is unsafe if files are ever
  untrusted.

Conclusion:

- This is one of the stronger modules in the project.


4.8 scripts/fetch_data.py

Strengths:

- Uses a realistic browser User-Agent.
- Has stronger-than-average scraper validation.
- Handles multiple PSX HTML layouts for KSE100 block extraction.
- Validates high/low/close/change/change_percent relationships.
- Archives the fetched result and keeps a latest copy.

Weaknesses:

- HTML scraping is inherently brittle if the PSX markup changes.
- No retry strategy or exponential backoff.
- No alerting if the HTML shape changes.
- "Open" is not actually scraped; it is proxied from previous close.

Professional judgment:

- This is a solid student-level scraper with validation.
- It is still not as reliable as using an official API or stable machine-
  readable feed.


4.9 scripts/clean_data.py

Strengths:

- Good required-column validation.
- Handles multiple date formats.
- Converts numeric fields robustly.
- Removes invalid rows and duplicate dates.
- Produces stable ascending output.

Weaknesses:

- Uses print + return None on failure instead of raising explicit exceptions.
- Drops to a generic failure path that can hide stack detail during debugging.
- Only required columns are preserved; extra source metadata is lost at this
  stage.

Conclusion:

- Functionally good, but error-handling style should be improved.


4.10 scripts/features.py

Strengths:

- Keeps a latest unlabeled row for API prediction.
- Produces useful beginner-level time-series features.
- Performs validation before writing output.

Weaknesses:

- Feature set is very basic.
- No gap-aware feature handling.
- No calendar/session-aware logic.
- No richer technical indicators or data quality profile.

Conclusion:

- Good as a first ML feature layer, but not enough for reliable market modeling.


4.11 scripts/train_model.py

Strengths:

- Saves model together with metadata, feature columns, latest row, warnings,
  metrics, and profile.
- Uses chronological split rather than random shuffling.
- Adds a useful warning when the latest date gap is large.

Weaknesses:

- The model is trained across a very large historical gap between the seed
  history and live row.
- Accuracy alone is not enough as a model quality metric.
- No walk-forward validation.
- No confidence/probability returned in API.
- No baseline comparison.
- No model registry/versioning.
- Still relies on local pickle files.

Professional judgment:

- The code is better than a typical beginner "just save model.pkl" script.
- The modeling approach is still weak for real financial prediction.


4.12 scripts/run_pipeline.py

Strengths:

- Clean orchestration of fetch -> clean -> feature -> train -> archive -> prune.
- Returns a useful summary object.
- Failure checks exist between stages.

Critical weakness:

- It always uses SEED_HISTORY_PATH as history input.
- That means multiple live runs do NOT build a cumulative live history.
- Old live rows are archived, but not preserved in the active training dataset
  unless they already exist in the seed file.

Why this is serious:

- The app appears to "hold old data", but the active dataset does not truly
  grow across live runs.
- This directly hurts model quality and contradicts the intended long-term
  pipeline behavior.

This is one of the most important issues to fix next.


4.13 static/index.html

Strengths:

- Strong visual improvement over a basic template.
- Tailwind dashboard is practical and modern.
- Trend chart and market structure chart are useful.
- Auto-refresh is implemented.
- File manager modal is a nice admin feature.

Weaknesses:

- Uses external CDN dependencies for Tailwind, Chart.js, and Google Fonts.
- That creates a runtime dependency on third-party networks.
- Snapshot UI still renders "Open" even though the model bundle no longer
  stores Open as a feature.
- No authentication flow around admin actions in the UI.

Conclusion:

- Good portfolio UI.
- Needs production hardening.


4.14 tests/test_psx_pipeline.py and tests/fixtures/psx_kse100_sample.html

Strengths:

- Good starting automated coverage.
- Tests scraper parsing.
- Tests cleaning, feature engineering, training, and file retention behavior.
- Tests protected file deletion and file listing route.

Weaknesses:

- Coverage is still narrow relative to the full app.
- No automated test for /predict or /insights behavior.
- No test for /api/pipeline/run concurrency.
- No frontend tests.
- No deployment/startup smoke test.
- No test proving the current-history accumulation bug.

Conclusion:

- Better than many student projects, but still far from comprehensive.

7. MAIN WEAKNESSES AND RISKS
========================================================================

CRITICAL ISSUES

1. Active live history is not cumulative.

- The pipeline merges the newest live file with the original seed file, not
  with the current accumulated history.
- This means prior live sessions are archived but not kept in the active
  dataset.

2. Historical data continuity is poor.

- The seed dataset ends years before the live row.
- Features such as Return, MA_7, MA_30, and Volatility become distorted when
  computed across such a large time gap.

3. Deployment from Git will not be self-sufficient.

- The repo does not track the seed CSV, processed files, or model file.
- A fresh deploy will not have the historical base file that the pipeline
  currently depends on.

4. Sensitive admin endpoints are unauthenticated.

- Anyone who can access the deployed app could trigger retraining or delete
  archive files.


HIGH-PRIORITY ISSUES

5. Long-running training is triggered inside an HTTP request.

- This can block the web worker.
- It makes timeouts and user-facing latency more likely.
- It is not a scalable production design.

6. Storage is tied to local files.

- This makes the app fragile across deployments, restarts, and scaling.
- It also limits deployment choices.

7. Procfile and dependencies are inconsistent.

- Procfile expects gunicorn.
- requirements.txt does not install gunicorn.

8. UI/admin operations depend on third-party CDNs.

- Tailwind CDN, Google Fonts, and Chart.js CDN are convenient for a demo but
  less ideal for hardened deployment.


MEDIUM-PRIORITY ISSUES

9. Error handling uses prints and None returns in several pipeline stages.

10. Docs are weak relative to project complexity.

11. Model evaluation is too shallow for financial forecasting.

12. Test coverage is good as a start, but not enough for deployment confidence.

11. WHERE I WOULD DEPLOY THIS WEB APP
========================================================================

My recommendation depends on your goal.

Best current platform for this project with the fewest conceptual jumps:

- Railway

Why Railway is a strong fit:

- Railway has a current official Flask deployment guide.
- Railway supports persistent volumes.
- Railway documents that if an app writes to a relative path like ./data,
  the volume should be mounted at the matching absolute path under /app.
- Railway gives a public HTTPS domain and has documented healthcheck support.
- Railway documents a maximum HTTP request duration of 15 minutes, which is
  relevant because your pipeline currently runs inside a web request.

Important caveat:

- I would still not keep long-term training inside a user-facing request.
- Even on Railway, I would eventually move retraining to a secure background
  or scheduled workflow.


Second strong option:

- Render

Why Render is good:

- Render has an official Flask deployment guide.
- Render supports persistent disks.
- Render supports web services, background workers, and cron jobs.

Why Render is slightly less natural for this exact design today:

- Render states that disks are only accessible by a single service instance.
- Render also states cron jobs cannot have attached disks.
- Because your current design relies on the same local files for the web app
  and pipeline outputs, that limitation becomes important.

Render becomes a better choice after one of these changes:

- the pipeline remains inside the same web service
- or the app is refactored to store state outside the local disk


Third option if you want the simplest "student project online quickly" path:

- PythonAnywhere

Why:

- It is friendly for Flask beginners.
- It gives persistent hosted filesystem behavior that fits your current local-
  file design better than many serverless-style platforms.

Why it is not my first long-term choice:

- It is less aligned with the more production-like architecture you are already
  moving toward.


My final deployment recommendation:

- For this codebase today: Railway first
- After storage/auth/job refactor: Railway or Render are both strong
