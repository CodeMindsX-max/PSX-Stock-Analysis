PSX Stock Analysis Project - Issues and Solutions
This document reviews the codebase file by file, identifies weaknesses and root causes, and proposes detailed fixes. Each issue is followed by clear solution steps and a Master Prompt to guide automated code fixes.
.gitignore
Issue: The .gitignore excludes critical data and model files (data/raw/*, data/processed/*, and *.pkl). This means the repository does not include the historical seed dataset or trained model. A fresh deploy from Git would lack the base data needed to run the app or build the model, leading to failures in data ingestion and predictions.
Solution:
* Include a seed dataset: Add a minimal historical seed CSV file to the repository (e.g. data/processed/seed.csv). This ensures a fresh clone has initial data. The file can be a small sample or compressed version of historical KSE100 data.
* Update pipeline logic: Modify the pipeline so that if the seed data is missing or outdated, it can be regenerated or fetched. For example, check at startup if data/processed/seed.csv exists; if not, populate it from an external source or a bundled sample.
* Adjust .gitignore patterns: Remove or modify patterns that exclude files you now want to keep. For instance, you might only ignore truly scratch data (like intermediate raw files) but keep one copy of processed data and the latest model in data/ or models/.
* Document setup: Update the README (or a setup script) to explain how to initialize or update the historical data. For example, include a command like python -m scripts.fetch_data to bootstrap data if needed.
Master Prompt:

Adjust .gitignore to no longer ignore the committed seed data and model. Add code in the data pipeline (e.g. run_pipeline.py) to check for missing historical data on first run and download or create it if absent. Provide instructions or code to bootstrap the historical dataset on a new deployment."

* Regular audit: Occasionally review requirements.txt for drift (use tools like pip-review or pipreqs).

app.py
Issue #1: Synchronous long-running pipeline. The /api/pipeline/run endpoint invokes the entire training pipeline inside the Flask request. This will block the web server until completion, leading to timeouts and a poor user experience.
Solution:
* Background task: Offload pipeline execution to a background job or separate process. For example:
* Use a task queue like Celery or RQ. The route would trigger a job and immediately return (possibly with a job ID to poll status).
* Alternatively, use Python's threading or multiprocessing to start the pipeline in a new thread or subprocess and return a quick HTTP response.
* Or, remove the pipeline trigger from the public API entirely and run it on a schedule (e.g., a cron job or platform scheduler).
* Immediate response: Change the endpoint to return a 202 Accepted (or similar) with a message that training has started.
Master Prompt:
"Refactor app.py so that /api/pipeline/run no longer executes the pipeline synchronously. Instead, integrate a background job system (like Celery with Redis/RabbitMQ or Python threading). When the route is called, queue the pipeline task and immediately return a success response. Ensure the front end handles this change (e.g., by polling an endpoint to check pipeline status)."
Issue #2: Lack of authentication/authorization. Endpoints that modify or delete data (/api/pipeline/run, /api/files/delete) are unprotected. Anyone with access to the app URL can trigger these destructive actions.
Solution:
* Add auth mechanism: Implement a simple authentication system. Options include:
* API tokens: Require a secret token or API key passed as a header or query parameter for admin routes.
* Session-based login: Create a login page and require login before showing admin controls. Use Flask-Login or Flask-Security for sessions.
* HTTP Basic Auth: For quick protection, use Basic Auth for admin routes.
* Protect routes: Decorate sensitive endpoints to require authentication. For example:

  @app.route('/api/pipeline/run')
@requires_auth
def run_pipeline():
    ...
* UI changes: Hide or disable the "Run Pipeline" button and file delete options unless the user is authenticated.
* CSRF protection: If using session forms, ensure CSRF tokens on form submissions (Flask-WTF can help).
Master Prompt:
"Secure the admin routes in app.py. Implement an authentication check on routes like /api/pipeline/run and /api/files/delete. For instance, add a decorator that verifies an admin API token or user session. Update the frontend to require login or include the token for admin actions. Provide code changes that enforce authentication on these endpoints."
Issue #3: Repeated disk I/O. The model and data files are reloaded from disk on every request (e.g. in the /predict endpoint). This is inefficient and may slow down responses.
Solution:
* Persistent loading: Load the model and necessary data once when the app starts, storing them in a global object. For example, load the model in the global scope or in an @app.before_first_request function.
* Cache feature columns: Instead of reading feature schema from file on each request, keep it in memory or embed it in the model metadata.
* Thread-safety: Ensure that the loaded model is thread-safe if using multiple workers (Gunicorn can pre-load).
Master Prompt:
"In app.py, change the code so that the machine learning model and feature metadata are loaded once at startup rather than on each request. For example, load model.pkl and feature columns into global variables or the Flask application context. Remove redundant file reads in each route. Test that predictions still work and that response times improve."
Issue #4: No structured logging. The application uses prints or minimal logging. This makes debugging and monitoring harder in production.
Solution:
* Use logging module: Configure Python's logging to output INFO/ERROR logs. For example, at the top of app.py:

  import logging
logging.basicConfig(level=logging.INFO, format='%(asctime)s %(levelname)s:%(message)s')
* Log important events: Replace print statements with logging.info(), logging.error(), etc., in routes and pipeline steps.
* External logging: If deploying to platforms, route logs to stdout so the hosting environment can capture them.
Master Prompt:
"Integrate Python's logging into app.py. Replace any print() calls with logging.info() or logging.error(). Configure a log format that includes timestamps and severity. Ensure that all key events (e.g., starting pipeline, serving a prediction, catching exceptions) are logged appropriately."
Issue #5: File system dependence. The app assumes a local filesystem for data persistence. This limits scaling (multiple servers would not share files) and makes deployments fragile if the container restarts.
Solution:
* Use environment configuration: Do not rely on hardcoded local paths. Instead, support environment variables for directories (covered more in pipeline_utils).
* External storage: For production, consider using a database or object storage (e.g. AWS S3, a managed SQL/NoSQL DB) for persistent state. At a minimum, ensure the platform has a persistent volume mount.
* Stateless design: Make the Flask app itself stateless by storing state (data, models) externally.
Master Prompt:
"Remove assumptions about local storage in app.py. Read data and model paths from environment variables or a config file. Explain how to configure persistent storage (e.g., mounting a volume). Consider integrating a database or remote storage for the data. Provide code changes and instructions for a stateless deployment."

scripts/train_model.py
Issue #1: Training across a large temporal gap. The model is trained on historical data that ends long before the latest live data. This can degrade performance because market conditions may have changed.
Solution:
* Walk-forward validation: Instead of a single train/test split, implement walk-forward or time series cross-validation. For example, split data into several contiguous blocks and train/test sequentially to better simulate real-world performance.
* Include recent data: Ensure the training set includes the most up-to-date labeled data. After each pipeline run, combine the new labeled row into training data so the model sees recent patterns. (Relates to pipeline fix in run_pipeline.py.)
* Retraining schedule: Retrain the model regularly (e.g. daily or weekly with the new data) to adapt to market changes. Document or automate this schedule.
Master Prompt:
"In scripts/train_model.py, change the validation strategy to use time-based cross-validation. For instance, split the historical data into multiple train/test splits that respect chronology. Update the training process so that after adding new data, the model is retrained including recent samples. Ensure that metrics (beyond simple accuracy) are computed and saved."
Issue #2: Limited evaluation metrics. The code only reports accuracy. For a stock direction model, accuracy alone is not sufficient (e.g. the baseline might be high). No probabilities or confidence scores are provided for predictions.
Solution:
* Additional metrics: Compute and save other metrics like precision, recall, F1-score, and especially a confusion matrix. For financial data, also consider profit-based metrics (e.g. strategy return).
* Probability output: If using a probabilistic classifier, modify the /predict API to return predicted probabilities along with class. This helps downstream users gauge confidence. Use model.predict_proba() and include that in the JSON response.
* Thresholding: Potentially allow different decision thresholds based on risk appetite.
* Baseline comparison: Train a trivial baseline model (e.g. always predicting "no change" or last known sign) and compare performance. Document whether the RandomForest beats this baseline.
* Persist metrics: Save all metrics and plots to the model metadata for auditing. For example, include a validation report (text or JSON) in the model bundle.
Master Prompt:
"Improve scripts/train_model.py by adding richer evaluation. After training, calculate metrics like precision, recall, F1, and add the confusion matrix. Also use predict_proba to store prediction probabilities. Modify the /predict API to return these probabilities or confidence. Finally, log or save all metrics to the model metadata so you can compare performance over time."
Issue #3: No model versioning or registry. The code overwrites model.pkl each run, losing any previous versions. There's no tracking of model versions or rollback capability.
Solution:
* Timestamped models: Save each trained model with a unique version or timestamp in its filename (e.g., model_20260419.pkl).
* Model registry: Store model metadata (version, training date, metrics) in a central registry (could be a simple JSON file or database).
* API uses latest model: Modify the API so it always loads the latest model, not a fixed model.pkl (or use a symlink current_model.pkl).
* Retention policy: Decide how many old models to keep and how to clean up (could reuse the archive/prune logic).
* Test loading: Ensure that after saving a new model, the app reloads the correct one.

Update the API loading logic to always use the newest model file. Record metadata (version, date, metrics) in a separate registry file. Ensure that old models are archived or pruned according to policy."
scripts/run_pipeline.py
Issue: Data history not cumulative. Currently, each pipeline run uses the original seed history file (SEED_HISTORY_PATH) and only appends the latest live row for that run. It does not accumulate historical live data. As a result, old live rows are saved only in archives but never included in the dataset used for training. This undermines the goal of continual learning.
Solution:
* Use processed history: Change the pipeline so that it reads the latest processed/cleaned data (which should include all previous runs) rather than the original seed every time. For example:
* After cleaning and merging, overwrite or update a master history file with the new data.
* In subsequent runs, use this updated history as the starting point.
* Append new data: Instead of always merging with SEED_HISTORY_PATH, merge data/processed/current_history.csv (which is updated each run).
* Archive strategy: Archive should still keep backups of each run (timestamped), but the "active" dataset should grow.
* Backfill process: If starting fresh, include logic to merge seed data and all accumulated live data so far.
* Testing cumulative logic: Simulate running the pipeline twice: after the second run, verify the processed dataset has two rows. Write a test or assertion to prevent future regressions.
Master Prompt:
"Revise scripts/run_pipeline.py so that the master dataset grows cumulatively. Instead of always reading from the original seed file, have it update the cleaned/processed history in place. For example, maintain a "current_history.csv" that gets appended with new live rows each run. Ensure that old data from previous runs is included in the training set and not just archived. Write code or tests to confirm that after multiple runs, the dataset contains all past data."
